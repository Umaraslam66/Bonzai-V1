## Verdict: PASS — by log decision AND recomputation

- **Decision (log):** job 1 = `fresh run`; job 2 = `resuming from stepstep=1275.ckpt`.
  This is the direct readout of `run_short`'s F8 branch (`resume_ckpt_path` returned the
  checkpoint, handed to `trainer.fit`). Not `fresh run` ⇒ the silent-restart-from-0
  failure mode did not occur.
- **Continuation (recomputation, not the log line):** `torch.load`'d the actual stored
  `global_step` — N = 1275 (the checkpoint job 2 resumed from), M = 1300 (job 2's first
  new checkpoint), highest = 2025. **M > N**, monotonic from the resume point. Lightning
  given `ckpt_path` restores `global_step` from the checkpoint; a restart would have
  reset to 0 and logged `fresh run`.

## Caveat surfaced (known limitation, not a failure)

Lightning warned: *"resuming from a checkpoint that ended before the epoch ended and your
dataloader is not resumable."* The model / optimizer / `global_step` continue correctly;
the **dataloader iteration order** is not restored on a *mid-epoch* resume, so the data
stream after a mid-epoch preemption is not bit-identical (a small slice may be re-seen or
skipped at the boundary). This is consistent with the earlier `ddp_resume_check`, which
proved **epoch-boundary** resume is functionally bit-identical (atol 1e-4). Acceptable for
the bake-off: it affects neither step accounting, checkpoint integrity, nor the resume
decision. Worth remembering when interpreting a long preempted run's exact loss curve.

## SEPARATE FINDING — `torch.compile` inductor crash (a step-(c) blocker, NOT a resume issue)

On the first attempt (before mid-training kills were in place), the job ran to completion
and **crashed in the post-train eval/generate phase** with a `torch.compile` inductor
**CPU codegen** failure:

```
g++ … CPU_CAPABILITY_AVX2 … cda4xc….cpp  → CalledProcessError exit 1
error: expected primary-expression before ')'
  auto tmp16 = decltype(tmp4)::blendv(tmp14, tmp4, tmp15.template cast<float,1>());
```

**Root cause:** `scaffold_ddp_short.sbatch` loads `cuda/12.2` but **no modern gcc**, so
inductor's CPU codegen invoked the system `g++` = **gcc-8.5.0** (visible in the include
paths), which cannot compile torch 2.5.1's AVX2 vectorized `blendv(...).template cast<>()`.
GPU training itself compiled fine (triton); only the CPU codegen in eval failed. This is
the **same toolchain class as the mamba `gcc/12.2.0` precondition.**

- It does **not** affect this resume proof (we kill before eval) or step (b) mamba (which
  already carries the gcc-12 precondition).
- It **does** block step (c) the scored bake-off, which needs eval/generate to produce
  scores. **Fix (on Umar's word, at step c):** `module load gcc/12.2.0` (+ `CC/CXX`) in the
  run sbatch so inductor codegen uses gcc-12 — mirroring the mamba precondition — or
  disable compile on the eval path. To be resolved before the first scored run.
