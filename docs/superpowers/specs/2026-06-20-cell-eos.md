# Design spec — cell-EOS (`<cell_end>` self-termination)

- **Date:** 2026-06-20
- **Branch:** `phase-2-cell-eos` (off `phase-2-bakeoff-2backbone`; this is a prerequisite to the scored bake-off — the model must self-terminate before generation cost is bounded). Redirect to `main` if preferred.
- **Status:** DESIGN — four review corrections folded in 2026-06-20 (terminator family / keep-260 / shape-first Tooth-1 / `(…,510,260)` Tooth-2) + Tooth 5 (generator-break isolation). CONFIRMED to implement; branch stays stacked off `phase-2-bakeoff-2backbone`. Implementation HALTs before the GPU cache-rebuild + Tooth-1 smoke. No scored run, no merge.

## 0. Problem (one paragraph)

The micro-AR generator has no stop signal: `generate_cell_tokens` runs `for _ in range(max_new)` with no `break`, and the training data carries no cell terminator (`<cell_end>` id 260 is `on_disk:false`, excluded from the vocab, and absent from every shard — confirmed 0/23 real cells). So every generated cell runs to the `max_new` cap (the 89M diagnostic emitted exactly 2048/2048 tokens on 64/64 cells), which makes the held-out realism eval cost 68–118% of the grant instead of ~3–5%. This spec teaches the model to emit `<cell_end>` at cell end and stop there.

## 1. Locked decisions (do NOT re-litigate — from prior investigation)

1. **Reuse id 260**, flip `<cell_end>` to `on_disk:true`. No new id → `n_subf_vocab` stays 1508 → head/embedding width unchanged → 53M param-match untouched.
2. **Inject at `build_shards_in_memory`** (cached, derivation-versioned), with empty-guard `(*toks, 260) if toks else ()` — NOT the in-code flatten append (post-cache; a 260-run and non-260-run would collide on the cache key — unacceptable for the locked 6-run matrix).
3. **No `max_len` bump** — the 260 counts inside the existing `DEFAULT_MAX_CELL_TOKENS=13312` budget. Measured: **0** train cells sit at body-length exactly 13,312, so this drops **zero** currently-kept cells (drop-not-clamp confirmed; 44 cells >13,312 already dropped, max 19,372).
4. **Bump `SHARD_CACHE_DERIVATION_VERSION` 1→2; rebuild caches** (fail-closed forces it, so no 260/non-260 mix possible).
5. **Generator: `break` on 260** in `generate.py`.

## 2. Verified facts the design rests on (citations)

- `load_sub_f_vocab()` (`src/cfm/data/sub_f/vocab.py:168-182`) concatenates five loaders — semantic / unknown / encoding_primitive / structural / boundary_reference — and **none read the `dataloader_sentinels` block**. So flipping `on_disk:true` in the YAML is **inert by itself**; a loader must pull 260 in (the fix adds a **sixth** loader `_load_cell_end_slot`, family `terminator` — see 3.A.2).
- `n_subf_vocab = max(vocab_tag_to_id().values()) + 1 = 1508` (max on-disk id 1507). 260 < 1507 → adding it does NOT change the max → head `nn.Linear(d_model, n_subf_vocab)` (`scaffold_backbone.py:88`) and embedding (`:64`) widths unchanged → param-match holds. `CONDITIONING_ID_BASE` (`conditioning.py:31`, same `max(...)+1`) also unchanged at 1508.
- The loss mask is **seq_len-relative** (`scaffold_backbone.py:156-162`): `target=ids[:,1:]`, mask `[:pl-1]` (prefix) and `[seq_len-1:]` (padding). Appending 260 (bumping `seq_len`) makes the `(last-510 → 260)` transition fall in the supervised window automatically, and 260→pad masked — **no mask-code change needed** (Tooth 3 proves this).
- Both KS sides split via `split_cell_into_features` (`seam_decodability.py:67-85`), which drops a trailing 260 (after the last 510, `cur=[]`). Real reference cells (no 260) and generated cells (with 260) yield identical per-feature values → **no KS bias**; the locked floor (sha `95abb88`) needs no re-extraction.

## 3. Exact changes (file:line)

Critically, a separate family lets the structural-range invariant (`{509,510}` / `509 ≤ id ≤ 510`) stay **honest** — folding 260 into `structural` would force relaxing that range, which **silently admits 261–508**. A `terminator` family is *less* churn than widening, not more (structural assertions are left entirely untouched; we add a parallel block — see A.3). *Fallback ONLY if `terminator` ever proves higher-churn than scoping: assert the explicit SET `{260,509,510}`, never a widened range — flag for confirmation, do not silently decide. (Not needed here.)*

### B. build_shards — inject 260 at non-empty cell-end (empty-guard)

5. **`src/cfm/data/training/build_shards.py:511`** — `tokens=tuple(toks),` → `tokens=(*toks, CELL_END_TOKEN_ID) if toks else (),`.
   - **Empty-guard is mandatory here** (unlike flatten): `build_shards_in_memory` builds a `CellPayload` for **every** cell incl. empties (`:506-517`, no drop — the empty-drop is downstream in `flatten` `datamodule.py:209`). A blanket append would turn `()` into `(260,)`. The `if toks` guard keeps empties `()`.
   - Over-length interaction: `flatten`'s `if n > max_cell_tokens` (`datamodule.py:212`) now sees the post-260 length; effective feature-body budget = 13,311. Measured impact: **0 cells** (none at exactly 13,312). No `max_len` change.

7. **`tests/training/test_shard_cache.py:300-305`** — `test_golden_fixture_pins_versions_to_the_cache_bytes`: update `"derivation": "1" → "2"` AND re-golden `cells_sha256` (changes because every non-empty cell's `tokens` now carries a trailing 260). **This is Tooth 4** (red-before / green-after). `tiles_sha256` unchanged. `:199` (`derivation_version == constant`) auto-passes.

### D. Generator — break on 260

## 4. TEETH (all must pass before trust — before any scored run)

> Ordering: A→B/C/D land + CPU teeth (2,3,4,5) green → cache rebuild → smoke-train tooth (1). Tooth 1 needs a FRESH train on the v2 cache; the existing diagnostic checkpoints were trained WITHOUT 260 and CANNOT be reused.

**Tooth 1 — gen EMITS 260 and UNPINS from the cap (GPU smoke).** After a short smoke train on the v2 (260) cache, generate N cells under matched conditioning. **PASS is SHAPE-FIRST, NOT a 95% emit-rate** — a sub-1-GPU-h smoke can underlearn a once-per-cell target, so a 95% gate risks a FALSE RED that is *undertraining, not broken wiring*. PASS requires BOTH:
- **(a) length distribution clearly UNPINNED from the cap** — p99 well below `max_new`, median in the low-hundreds-to-~600 band (the real ~600 region), NOT a spike at `max_new`; AND
- **(b) emit-rate materially > 0** — a decisive contrast against the pre-fix baseline (0% emit / 100% of cells at the cap).

Together these prove the wiring learned to stop. **Reserve ≥95% emit for the SCORED runs** (a fully-trained model), not this smoke. **Smoke budget: sub-1-GPU-h `fast_dev`** (NOT a full run); the existing diagnostic checkpoints were trained WITHOUT 260 and CANNOT be reused.

**Tooth 2 — rebuilt cache: byte-deterministic, 260 only at non-empty cell-end, ends `(…,510,260)`, 0 empties carry it.**
- `tests/training/test_shard_cache.py:91` `test_city_cache_bytes_are_deterministic` — re-run on v2, GREEN (determinism preserved).
- NEW data-integrity check over the rebuilt cache `tokens`: for every cell — `tokens[-1] == 260` **iff** non-empty; `tokens.count(260) == 1` for non-empty (260 never interior: `260 not in tokens[:-1]`); empty cells `tokens == ()` (no 260); **AND every non-empty cell ends exactly `(…, 510, 260)`** — i.e. `tokens[-2] == 510` (the last feature's `<feature_end>` immediately precedes the cell terminator). This `(510, 260)` assertion proves the real cell-ends match the `510 → 260` transition that Tooth 3 supervises on a **synthetic** fixture — so Tooth 3 is testing a REPRESENTATIVE cell-end, not an assumed one. Run as a fixture-level unit test on `build_shards_in_memory` output AND a spot-scan of the real rebuilt cache.

**Tooth 3 — loss supervises the 260 target.** Unit test: construct a `CellExample` whose body ends `…, 510, 260`; call `training_loss`; assert (a) the next-token target at the final-510 position **== 260** and is **not** `_IGNORE` (it contributes to CE; `n_supervised_positions` counts it); (b) the 260's own next-target (first pad) **is** masked. Proves the model is trained to emit 260 with no mask-code change.

**Tooth 4 — golden fixture regenerated for derivation v2 (red-before / green-after).** Before re-goldening: with `SHARD_CACHE_DERIVATION_VERSION="2"` + the build_shards change, `test_golden_fixture_pins_versions_to_the_cache_bytes` is **RED** (both `derivation` "1"≠"2" and `cells_sha256` mismatch). After updating the asserted `derivation→"2"` and the new `cells_sha256`: **GREEN**. The red phase is the proof the bump+format change were not silent. *Capture both phases in the PR/report.*

**Tooth 5 — generator BREAK isolated from model learning (CPU, no GPU/no train).** Proves the D.9 `break` wiring on its own, decoupled from whether any model *learned* to emit 260 — so a future Tooth-1 RED can be triaged. Construct a **controllable stand-in** for `MicroAR` exposing the interface `generate_cell_tokens` touches (`.training`, `.eval()`, `.train(mode)`, `.parameters()` yielding ≥1 tensor for the device probe, and `__call__(ids, char_stats=…)` returning logits of shape `(1, n_subf_vocab)`) — e.g. a tiny real `MicroAR` with `forward` monkeypatched, or a minimal stub. Drive its logits to force a **non-260** id for the first `k-1` sampled steps and **260** on the `k`-th step (one-hot-ish logits: favored id `+large`, rest `-large`, so `multinomial` picks it with prob ≈ 1 — deterministic). Call `generate_cell_tokens(model, prefix=…, max_new=M, seed=…)` with **`M ≫ k`**. Assert:
- (a) the returned tail has length **exactly `k`** — it stops on the step that emitted 260, NOT at `max_new`;
- (b) `tail[-1] == 260` — the trailing 260 is **included** (D.9-keep);
- (c) `len(tail) < max_new` — decisively unpinned from the cap; and as a guard, the stub's `__call__` was invoked exactly `k` times (no over-run past the break).

Triage value: **Tooth 5 GREEN + Tooth 1 RED ⇒ undertraining, not broken break-logic** (the wiring is proven; the smoke model just hasn't learned the once-per-cell target). Tooth 5 RED ⇒ the `break` itself is wrong, independent of any training.

## 5. Non-goals / out of scope

See A.2 / A.3. *Fallback if it ever proved higher-churn: explicit SET `{260,509,510}`, flagged not silently widened — not needed (terminator is the lower-churn path).*
2. **Generator KEEPS the trailing 260** (mirrors training `…510,260`; `split_cell_into_features` drops it). See D.9.
3. **Tooth-1 PASS is shape-first** — length unpinned from the cap (p99 ≪ max_new, median low-hundreds-to-~600) AND emit-rate materially >0 vs the 100%-at-cap baseline — NOT a 95% gate. ≥95% emit is reserved for the scored runs. See Tooth 1.
4. **Tooth-1 smoke budget: sub-1-GPU-h `fast_dev`** (the one GPU gate; teeth 2/3/4 are CPU). See Tooth 1.

## 7. Risk register

- **Vocab count drift:** `SUB_F_ON_DISK_TOTAL` + the A.3 test sites (total `687`, the new `terminator` block, the sentinel-exclusion loop) must move together (A.3 guards). A miss goes red in `test_vocab.py`.
