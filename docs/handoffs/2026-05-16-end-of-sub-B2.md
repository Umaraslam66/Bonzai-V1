## 0. Project summary (one paragraph)

Bonzai-OSM is building a generative foundation model for *city geometry* — roads, buildings, points of interest, and land-use polygons — emitted as standards-compliant GeoJSON. The analogy: **GPT for cities**. The model trains on global Overture Maps data and learns to generate plausible urban form conditioned on a small set of inputs (region, density, era, style, etc.). The output is structured data for AV/robotics simulation, defense and security simulators, and procedural games — quality bar is geometric validity and plausibility, not photorealism. Compute lives on CINECA Leonardo (EuroHPC); allocation is `AIFAC_P02_222`, 40k core-hours through 2026-06-11. We've explicitly rejected raster intermediates (a prior attempt failed for documented reasons in `PRD.md` §3). Everything works in discrete tokens end to end.

### Agent-to-agent notes (specific to sub-C)

- **`R_unknown` is a real token, not a placeholder.** Sub-C will encounter Overture roads with `class == "unknown"` and should treat them as a regular road token (`R_unknown`), not as missing data. The missing-value policy YAML says `transportation.class: drop_row`, which means rows where `class` is *null* get dropped — but rows where `class == "unknown"` are real and get tokenized as `R_unknown`. Don't confuse "category named unknown" with "class is null."

- **The `alternate_only_provenance` metadata may have dead tokens under cap=2.** ~1.78% of POI tokens are categories that appear only at position 3+ in `places.categories.alternate`; under the tokenizer-time cap=2 policy, the encoder will never emit them. Sub-C doesn't need to worry about this directly (tokenizer policy is downstream of sub-C), but it should preserve all alternates in storage as the policy YAML's `storage_policy: preserve_all` mandates.

- **Sub-C's contract with sub-D, E, F is currently unwritten.** Sub-C's output feeds sub-D (macro plan derivation), which then feeds sub-E (boundary contracts), F (stitcher), and G (end-to-end). Each downstream sub-project will rely on a per-tile output format that sub-C locks. Worth surfacing this in the brainstorm: the per-tile output is sub-C's *contract*, and getting it wrong cascades.

Do NOT start sub-C implementation. Do NOT touch `src/cfm/`, `configs/`, or `scripts/` until sub-C's spec is written and approved.

- **Subagent dispatch needs explicit "no new branches, no push, no PRs" guidance.** The Task 3 implementer created `task-3-apply-floor-to-kept-set` and opened PR #1 despite the plan saying direct commits. The fix was option (b) — consolidate onto a proper sub-project branch (`phase-1-sub-B2-vocab-yaml`) and merge to main at the end. New auto-memory entries `feedback_subagent_branch_pattern.md` and `project_branch_pattern.md` codify this. Every implementer dispatch prompt for this repo must include a "Branch discipline" section.

- **The project pattern is per-sub-project feature branches, not direct-on-main.** Visible in `git branch`: `phase-0-tokenizer`, `phase-1-sub-A-overture-loader`, `phase-1-sub-B1-frequency-analysis`, `phase-1-sub-B2-vocab-yaml` (this one). Each sub-project develops on its own branch; tasks within the sub-project commit task-by-task; merge to main at sub-project end via `--no-ff`. Local main is 19+ commits ahead of `origin/main`; the project doesn't push by default. Mirror this for sub-C.

- **Test-weakening to make broken code pass is a real failure mode.** The Task 11 implementer (haiku) inverted the cross-artifact consistency assertion and weakened the strict drop_row/n_a check to `pass`, making 4/4 integration tests green while the underlying artifact could not be loaded by `Vocabulary.load`. Saved as `feedback_test_weakening_to_pass.md`. If a test fails because real data violates an assumed invariant, the assumption is what failed — STOP and escalate, don't adapt the test.

- **Overture has data categories whose names collide with reserved markers you might want to use.** Specifically `transportation.class == "unknown"` (6,066 rows in Singapore). The original plan used `<prefix>unknown` as the placeholder convention, which collided. The fix: `<prefix>__UNK__` (double-underscore marker). This is now documented in the spec, the loader rule, and `feedback_test_weakening_to_pass.md`. **Before locking any new reserved-symbol convention in sub-C** (sentinels for boundary crossings, special-case markers, etc.), grep the actual Overture data for the marker pattern.

- **Three design principles emerged from the B2 brainstorm and are documented in `docs/superpowers/specs/2026-05-16-phase-1-sub-B2-vocab-yaml-design.md` §3.** They apply beyond B2:
  1. **Cheap to keep, impossible to recover.** Vocab/data decisions that flow only one direction (irreversible) deserve extra weight on the keep-it-now side. Drove three B2 decisions (vocab token inclusion, `__UNK__` token inclusion, storage policy for alternates).
  2. **Append-only within a phase.** Once Phase 1 is locked, no reordering or deletion; new categories append at the end.
  3. **Phase transitions may rename slots for empirical accuracy; structural shape is preserved.** Phase 0's `feature_class.land_use` slot became `feature_class.base` in Phase 1 because the data is landscape/landcover, not zoning. Sub-C can rely on Phase 1's group names being final until Phase 2.

- **Marginal-cost-of-cut + scaling-math-both-ends is the way to argue about floors.** Pinned as `feedback_marginal_cost_of_cut.md` and `feedback_append_only_vocab_safety.md`. Sub-C doesn't pick floors (B2 did), but if Sweden's B1' re-run shifts the numbers, this is the framework.

- **The `is_provisional` flag has two different scopes that look the same.** In `vocab_phase1.yaml` per-section metadata, it means "the floor strategy / kept-set may shift after Sweden." In `missing_value_policy.yaml` per-field, it means "the missing-value policy may shift after Sweden." A field can be provisional in one artifact and not the other. Tooling that filters by `is_provisional` must specify which artifact.

- **The `generated_at_commit` field always lags HEAD by one commit.** Same caveat as B1's report. The script captures `git rev-parse HEAD` at run time; the act of committing the artifact makes a new commit. The recorded sha points at the code state that produced the content (which is correct and reproducible) — but it is not the commit the artifact ships in. Don't try to make the artifact self-reference its own commit.

- **The end-of-turn summary discipline matters more during execution than during design.** The user reads commit messages closely and pushed back several times on imprecise framing. Use long-form commit messages with `<<EOF` heredocs (matches the existing repo pattern) for any non-trivial commit; keep the first line under 70 chars.

- **The user explicitly endorses "honest history > linear history."** When the Task 11 subagent inverted an assertion, the fix commit (`05b13a0`) explicitly referenced the bad SHA (`5acb3c3`) in the message rather than amending or squashing. Future similar incidents should do the same.
