## 0. Project summary (one paragraph)

Bonzai-OSM is building a generative foundation model for *city geometry* — roads, buildings, points of interest, and land-use polygons — emitted as standards-compliant GeoJSON. The analogy: **GPT for cities**. The model trains on global Overture Maps data and learns to generate plausible urban form conditioned on a small set of inputs (region, density, era, style, etc.). The output is structured data for AV/robotics simulation, defense and security simulators, and procedural games — quality bar is geometric validity and plausibility, not photorealism. Compute lives on CINECA Leonardo (EuroHPC); allocation is `AIFAC_P02_222`, 40 k core-hours through 2026-06-11. We've explicitly rejected raster intermediates (a prior attempt failed for documented reasons in `PRD.md` §3). Everything works in discrete tokens end to end.

### Agent-to-agent note: `divisions.country` is probably not a vocabulary field

When B2 reads §3.9 of the report it will see `divisions.country`: 5 distinct values, 729 rows, one dominant value (Singapore itself). The mechanical thing to do is apply a floor strategy and end up with 1-5 tokens in the vocab.

**Don't do that without thinking.** `divisions.country` describes the country a division geometry sits in. Its real role is more naturally a **conditioning input** (region tag in the conditioning vector at the start of every training sequence per PRD §8) or a **hard-coded enum** of allowed countries we train on, not a learned token in the geometry vocabulary. The token-vocabulary is for geometric primitives and feature classes; the conditioning vocabulary is for "what kind of city are we generating." Country is conditioning.

Flag this explicitly to the user before B2 decides. Worth surfacing because it's the kind of subtle scope decision that gets lost when mechanically iterating through a list of fields.

Do NOT start B2 implementation. Do NOT touch `src/cfm/`, `configs/`, or `scripts/` until B2's spec is written and approved.

## 7. Things I (this session) wish I'd known at the start

- **`pc.struct_field` vs `chunk.field(i)` for pyarrow struct navigation.** The original plan used `chunk.field(field_index)` to extract a struct child column. That returns empty strings instead of nulls when the parent struct row is null, which over-counts `n_present`. The fix is `pyarrow.compute.struct_field(column, key)` — it propagates struct-level nulls correctly. The B1 implementer caught this during TDD on Task 3; the fix and the reason are documented inline at `src/cfm/data/frequency.py:103-105`. If you find yourself navigating pyarrow structs in B2, use `pc.struct_field`.
- **The report's `commit_sha` always lags HEAD by one commit.** The script captures `git rev-parse HEAD` at run time, but the act of committing the report makes a new commit. So the report file at commit `0b0df40` records `commit_sha=9736164`. This is fine — `9736164` is real, mergeable, and points at the code state at the moment the data was analysed. Don't try to make the report self-reference its own commit; it can't.
- **`Path.cwd()` for repo-relative paths is brittle.** B1's path-fix commit (`69cbd74`) uses `region.manifest_path.relative_to(Path.cwd())` with a `try/except` fallback. This works because the script is invoked from repo root. If you ever invoke the script from elsewhere, you'll get an absolute path — the fallback. The user spotted the original absolute-path bug; spot-check generated artefacts for machine-specific paths before committing.
- **Determinism check is sensitive to where you re-run from.** The plan's Step 9.5 re-runs the script after committing the report, which updates `commit_sha` and `Run UTC` in the rewritten report. The integration test isolates outputs to `tmp_path`, but a production-style determinism check needs to either (a) restore the report from git after the re-run or (b) accept that re-running mutates `commit_sha`. The B1 plan does (a). Document the choice in your plan.
- **Matplotlib's PNG metadata strip works but is best-effort.** The script passes `metadata={"Software": "", "Creator": ""}` to `savefig`. This handles the obvious version-drift footer but doesn't guarantee byte-identity across matplotlib patch releases. The spec explicitly documents this as an acceptable exception (§7 "Determinism caveat"). Don't burn cycles trying to make it perfect.
- **Sub-A's `LocalFixtureBackend` is effectively ignored when a cache exists.** `load_region` cache-hits before consulting the backend parameter. B1's integration test passes `--backend fixture` but actually reads real cached Singapore data because the cache is present. The test is still correct (shape-only assertions) but the test name is misleading. If B2 needs true backend isolation, it has to either delete the cache for the test (don't — wastes the user's 8 hours) or pass `refresh=True` (which would trigger a fetch from `LocalFixtureBackend`). Easier: hand-craft test data and call library functions directly instead of going through `load_region`.
- **The B1 implementer subagents were all `haiku` or `sonnet` — no opus needed.** The plan was specific enough that mechanical implementers handled it cleanly. The two-stage formal review was largely skipped in favor of controller verification because each task was self-contained TDD. This matches sub-A's pattern. Use the same judgment for B2.
- **The user reads end-of-turn summaries; long ones get skimmed.** Headers and tables help. One or two sentences plus a tight table when there's tabular state is the right level.
- **The user's review questions are sharp.** B1 got two real corrections from review: PRD-framing flag for Very strict (§6 of spec) and the absolute-path bug (last-minute, post-implementation). Plan for the user to catch issues you missed; don't get defensive.
